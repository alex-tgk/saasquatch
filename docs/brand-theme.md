**SaaSQuatch** is an enterprise-grade CLI tool that empowers startups and small teams to build production-ready microservices architectures in minutes. We provide comprehensive, batteries-included solutions with the power and flexibility of enterprise tools, delivered with a helpful, guiding approach.

### Brand Positioning

**"Enterprise-Grade Tools for Ambitious Startups"**

SaaSQuatch bridges the gap between quick scaffolding tools and enterprise frameworks. We believe startups deserve the same level of quality, security, and best practices as Fortune 500 companiesâ€”without the complexity, cost, or learning curve.

### Brand Promise

**Complete. Powerful. Approachable.**

- **Complete**: Everything you need, nothing you don't. Full-stack microservices architecture out of the box.
- **Powerful**: Enterprise-grade patterns, security, and scalability built-in from day one.
- **Approachable**: Clear documentation, helpful errors, and smart defaults that guide you to success.

## Target Audience

### Primary: Startups & Small Teams (2-10 developers)

**Profile**:
- Moving fast, need to ship features quickly
- Care about quality but don't have time for extensive research
- Want enterprise-level practices without enterprise complexity
- Value good defaults and clear guidance
- Tech-savvy but appreciate not reinventing the wheel

**Pain Points We Solve**:
- "How do I structure a microservices app correctly?"
- "What are the production best practices for Fastify?"
- "How do I set up multi-tenancy properly?"
- "What's the right way to handle authentication?"

### Secondary: Individual Developers

Solo developers and technical founders who want to build production-grade apps from the start.

## Content Guidelines

### Writing Style

**Principles**:
1. **Clarity over cleverness** - Say what you mean, directly
2. **Active voice** - "SaaSQuatch generates..." not "Projects are generated..."
3. **Present tense** - "The CLI creates..." not "The CLI will create..."
4. **Second person** - "You can configure..." not "Developers can configure..."
5. **Specific over generic** - "JWT authentication with @fastify/jwt" not "security features"

### Documentation Tone

- **Confident but humble** - We know what works, but we're here to help, not lecture
- **Explanatory** - Don't just say what, explain why
- **Anticipatory** - Address common questions before they're asked
- **Encouraging** - Celebrate progress, acknowledge challenges

**Example - Good**:
> Your auth service is now running with JWT authentication. The default secret is fine for development, but you'll need to set `JWT_SECRET` for production. Generate a secure secret with:
> ```bash
> openssl rand -base64 32
> ```

**Example - Bad**:
> Auth service started. Warning: Using default JWT secret.

### Error Messages

## Brand Do's and Don'ts

### âœ… DO

- Use Steel Blue and Royal Blue as primary brand colors
- Maintain consistent typography (Inter + JetBrains Mono)
- Use helpful, guiding language in all communications
- Show real code examples with proper syntax highlighting
- Emphasize production-readiness and best practices
- Provide clear, actionable error messages
- Use emoji sparingly for visual emphasis (âœ“ âœ— ğŸ’¡ ğŸš€)
- Maintain white space and clean layouts

### âŒ DON'T

- Use overly bright or neon colors
- Mix too many font families (stick to Inter + JetBrains Mono)
- Use condescending or gatekeeping language
- Show incomplete or "TODO" code examples
- Overpromise or use hyperbolic marketing speak ("revolutionary", "game-changing")
- Use angry or aggressive error messages
- Overuse emojis (no emoji soup)
- Clutter interfaces with unnecessary elements
